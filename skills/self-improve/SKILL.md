# Self-Improve — keep vibe-stack honest about the ecosystem

A periodic loop that compares vibe-stack against the moving target of the AI-coding-agent
orchestration space and writes the result down where the team will see it. It does **not**
change code — it produces a **decision input**: a dated GitHub Issue with two columns,
*what others do better* and *what we do uniquely*. The human decides what to act on.

> This is the **judgement layer**: it reads our own skills, reads the competition via the
> GitHub API, and renders a verdict. Pair it with any repo-discovery primitive you have (e.g.
> a `vibe-watch-ecosystem` watcher) to feed it candidates, but it stands alone on `gh` below.

---

## When to run

- On a cadence (weekly is plenty — the space moves fast but not daily).
- Before a planning session, to seed "what should we build next?"
- Whenever an ecosystem watcher surfaces a new repo worth a closer look.

Run it from the repo root: the orchestrator (Hermes) executes the steps below and reports the
Issue URL back to the user. Never go silent — surface the Issue link when done.

---

## What it does (4 steps)

### 1. Read our own surface area
Read **every `SKILL.md` in the repo** so the comparison reflects what we actually ship today,
not last month's memory.

```bash
# Every skill we ship, newest content first.
find . -name SKILL.md -not -path './node_modules/*' -not -path './.git/*'
```

Read each one. Build a short internal list of vibe-stack's distinctive claims (chat-first
control, portable markdown skills, lanes/race-mode/merge-captain, the `vibe` CLI, proactive
status reporting). These are the "what we do uniquely" candidates.

### 2. Scan GitHub for fresh competition
Search the GitHub API for orchestration projects that are **actively maintained** (pushed in
the last 7 days) and have **traction** (stars > 50):

```bash
# Query the API for the comparison set (bash 3.2-safe; BSD date then GNU fallback).
since=$(date -v-7d +%Y-%m-%d 2>/dev/null || date -d '7 days ago' +%Y-%m-%d)
for q in "tmux agent orchestration" "claude code orchestrator" "vibe coding agents" "coding agent dashboard"; do
  gh search repos "$q stars:>50 pushed:>=$since" --limit 20 \
    --json fullName,stargazersCount,description,url,pushedAt
done
```

For each notable repo, `gh repo view <owner/name>` or WebFetch the README and extract the
concrete features (sandboxing, diff-review UI, push notifications, model routing, MCP
lifecycle, setup hooks, review gates, embedded browser, …).

### 3. Judge — two columns
Distil the scan into two honest lists. Be specific and cite the repo for each claim.

- **What others do better** — features/patterns a competitor ships that we don't. Name the
  repo. This is the steal-list. (Recurring examples to check for: visual diff-review /
  approve-before-merge UI, push-on-"needs-input" notifications, structured JSON status with
  automatic state inference, mid-task redirect to a live agent, per-task model-cost routing,
  sandbox tiers beyond worktrees, worktree setup-hooks, review gates, embedded scriptable
  browser, MCP lifecycle management.)
- **What we do uniquely** — what vibe-stack still does that nobody else does, or does better.
  (Today: chat-first / channel-agnostic control with no port or VPN; portable one-line
  markdown skills that *compose* with the other tools instead of replacing them; and the
  open whitespace — a single in-place-updating chat message rendering the whole fleet's live
  status, plus interval heartbeats, which almost no orchestrator does.)

Honesty rule: if a competitor genuinely does our "unique" thing now, **move it to the other
column.** The value of this report is that it doesn't flatter us.

### 4. File the Issue
Open a GitHub Issue on the repo with a dated title and the two sections:

```bash
DATE=$(date +%Y-%m-%d)
gh issue create \
  --repo woodylin0920-bit/vibe-stack \
  --title "Self-improvement report: $DATE" \
  --body "$(cat <<'EOF'
## What others do better
<!-- one bullet per competitor feature we lack; cite the repo -->
- ...

## What we do uniquely
<!-- one bullet per durable differentiator; drop anything a competitor now matches -->
- ...

---
_Generated by the `self-improve` skill. Scan window: repos pushed in the last 7 days, stars > 50._
EOF
)"
```

Report the Issue URL back to the user. The Issue is the deliverable — the next planning pass
decides which steal-list items become work.

## Guardrails

- **Read-only on the codebase.** This skill never edits skills or code; it produces an Issue.
  Acting on the findings is a separate, human-approved step.
- **Cite every "others do better" claim** with the repo it came from — an uncited claim is a
  rumour, not a decision input.
- **Keep it tight.** A handful of high-signal bullets per column beats an exhaustive dump;
  the goal is decisions, not a literature review.
- **Stay honest.** The ecosystem map ([docs/ecosystem.md](../../docs/ecosystem.md)) and the
  README's comparison table are the things this report keeps current — if the report
  contradicts them, the docs are stale and should be updated.

## Pitfalls

1. **`gh` not authenticated** → the search returns nothing. Check `gh auth status` first.
2. **`date -v` vs `date -d`** — macOS (BSD) uses `date -v-7d`, Linux (GNU) uses
   `date -d '7 days ago'`. The snippets above try BSD then fall back to GNU. bash 3.2 safe.
3. **Star/recency filters hide good small tools** — the thresholds are a noise filter, not
   truth. If a watcher flags something below them as interesting, include it.
4. **Flattery** — the most common failure is a "what we do uniquely" list that's really a
   wishlist. Only list things that survive checking a competitor's current README.
